# Thread opt-out for lwchat — design

**Date:** 2026-06-04
**Status:** Approved for planning

## Problem

lwchat posts to Google Chat threads on an agent's behalf. A thread's
participants have no way to tell lwchat to stop. We want a self-service,
thread-scoped mute: anyone in a thread can reply with a hashtag and lwchat
will stop posting in that thread.

## Behavior summary

When the protection is enabled (default):

1. **Footer** — every `reply` and `post --thread` message gets an
   auto-generated footer telling people how to opt out.
2. **Opt-out check** — before sending, the CLI scans the target thread. If
   anyone has replied with a message that is **exactly** the opt-out hashtag,
   the CLI refuses to send and exits with a clear opt-out error.
3. **Skill guidance** — `SKILL.md` documents the behavior so the agent
   understands the refusal and explains it to the user.

A single config toggle governs both halves, so the protection can be turned
off entirely in the future without code changes.

### Out of scope

- New top-level `post` (no existing thread to opt out of) and `dm`.
- Any "opt back in" command.
- A per-call `--force` override (the config toggle is the only escape hatch;
  a per-call bypass would let an agent trivially defeat the opt-out).

## Config

Add one block to `DEFAULT_CONFIG` in `lib/config.js`:

```js
thread_optout: {
  enabled: true,          // master switch — false = behave exactly as today
  hashtag: "#stoplwchat", // what people reply with; also what the scan matches
},
```

- `enabled: false` disables **both** the footer and the scan.
- `hashtag` is the single source of truth: the footer text and the scan both
  read this string, so they can never drift apart, and the tag can be renamed
  later without touching code.
- Existing installs inherit `enabled: true` automatically — `loadConfig`
  already merges the user's config over `DEFAULT_CONFIG`, so a missing
  `thread_optout` key takes the default.

## Footer

Appended to the message text **after** mention resolution (the footer has no
`@mentions`, so it must not go through that pass). Applied on every
`reply` / `post --thread` message:

```
<message>

_(Auto-generated by lwchat — reply #stoplwchat to mute lwchat in this thread.)_
```

- The hashtag shown comes from `config.thread_optout.hashtag`.
- Italic markdown (`_..._`) renders in Google Chat.
- The footer template lives in code; only the hashtag is config-driven.

## Opt-out check

Helper: `threadHasOptOut(spaceId, threadName, hashtag)`

- Uses the existing `listThreadMessages(spaceId, threadName)`.
- A message counts as an opt-out **only if**
  `message.text.trim().toLowerCase() === hashtag.toLowerCase()`.
- Returns `true` on the first such message, else `false`.

### Why exact-match, not substring

Multiple people may run lwchat under their own Google accounts. Each one's
footer is posted under *their* user id and contains the hashtag. A
substring match (or a "skip my own messages" exclusion keyed on our own
user id) would therefore treat another user's footer as an opt-out and
falsely mute the thread.

Requiring the message to **equal** the hashtag exactly avoids this entirely:
a footer is always a full sentence and can never equal `#stoplwchat`, no
matter who posted it. This also removes the need for a `getMe()` lookup —
the helper needs no knowledge of our own identity.

### Match strictness

Trim surrounding whitespace, lowercase both sides, require exact equality.
`#stoplwchat` and `  #StopLwchat ` match; `#stoplwchat please` and
`#stoplwchat.` do **not**. (Trailing-punctuation tolerance was considered and
rejected — strict equality matches the intent literally.)

## Wiring

- `cmdReply` (`lib/commands.js`): always threaded. Before
  `sendMessage`, when `thread_optout.enabled`, run the check on
  `entry.space` / `entry.thread`; on hit, refuse. Append footer to the
  resolved text before sending.
- `cmdPost`: run the check and append the footer **only when `--thread`
  is given** (a brand-new top-level post has no thread to mute). When
  posting top-level, behave as today.
- `cmdDm`: unchanged.

### Refusal output

- Non-JSON: `console.error` a clear message
  (e.g. `Thread opted out of lwchat (#stoplwchat). Not posting.`) and exit
  non-zero.
- JSON: `{ ok: false, error: "...", opted_out: true }` so the agent can
  detect the reason programmatically and report it to the user.

## SKILL.md

Document under the `reply` / `post` sections:

- Each posted message carries the auto-generated opt-out footer.
- A thread is muted when someone replies with a message that is exactly
  `#stoplwchat`; lwchat will then refuse to post there and return
  `opted_out: true`.
- The behavior is governed by `config.thread_optout` and can be disabled.

## Testing / verification

- `threadHasOptOut` returns `true` for a thread containing a lone
  `#stoplwchat` (and case/whitespace variants), `false` for a thread where
  the hashtag only appears inside a footer or a longer sentence.
- With `enabled: true`, `reply`/`post --thread` append the footer; with
  `enabled: false`, output is byte-identical to today (no footer, no scan).
- A reply to an opted-out thread exits non-zero (text) /
  `{ ok: false, opted_out: true }` (json) and sends nothing.
- Top-level `post` (no `--thread`) and `dm` are unaffected.
