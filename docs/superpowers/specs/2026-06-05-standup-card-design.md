# `lwchat standup --card` — design

**Date:** 2026-06-05
**Status:** Approved for planning
**Builds on:** `docs/superpowers/specs/2026-06-05-standup-design.md` (the base `standup` command)

## Problem

The text/JSON `standup` output is functional but plain. We want the standup to
appear as a rich, **clickable** card in a Google Chat space (the concept the user
mocked up): a header, a section per bucket with counts, one row per issue with
the college, a clickable issue id (→ Redmine), the subject (→ thread), the action
time, and a colored status chip — plus a summary.

Google Chat renders this via **`cardsV2`**, which is rejected for human-OAuth
credentials (what lwchat uses for its other commands). The way to post a card
without a server or service account is a **Chat incoming webhook**: a Google-
hosted URL you `POST` a `cardsV2` payload to. This was proven end-to-end against
the real Daily-Summary space during design.

## Behavior

```
lwchat standup --card [--webhook <alias|url>] [--hours N] [--space <alias>] [--json]
```

- Runs the same scan/classification as `standup`, then builds a `cardsV2` payload
  and `POST`s it to a Chat incoming webhook. The card renders natively and its
  links are clickable — no image, no server.
- Without `--card`, `standup` behaves exactly as today (text or `--json`).
- **Default window changes from 30h to 24h** for the whole `standup` command (the
  30h overlap was a fudge; 24h matches the daily cadence). `--hours N` overrides
  (e.g. `--hours 72` on Mondays / after holidays).

### Webhook resolution

Webhook URLs are secrets and live only in `~/.lwchat/config.json` under
`standup_webhooks` (alias → url); never in the repo.

- `--webhook <url>` (starts with `http`) → use directly.
- `--webhook <alias>` → look up `config.standup_webhooks[alias]`.
- neither → if `standup_webhooks` has exactly one entry, use it; otherwise fail
  with a clear message listing the configured aliases.

### Out of scope

- Posting the card via human-OAuth / the normal `post` path (impossible —
  cardsV2 is blocked there).
- Interactive in-card buttons that call back into our code (would need a hosted
  endpoint). Links that open Redmine/threads need no backend and are included.
- Auto-scheduling the daily run (that's a cron / the `/schedule` skill, separate).
- Weekday/holiday-aware windows (decided against — `--hours` covers it).

## Card layout (finalized in design against the live space)

- **Header:** title `Daily Standup — <me name>`, subtitle
  `<DD Mon YYYY> (last <N>h) · Generated by LWChat Standup Bot`.
- **One section per non-empty bucket**, in `BUCKET_ORDER`, header
  `<emoji> <Label> (<count>)` — e.g. `🚀 Released to Production (6)`.
- **One `decoratedText` row per item**, two lines:
  - **Line 1:** `<b>college</b> · <a href="issue_url">#id</a> · <font color="#888888">time</font>`
    (college / id / time each omitted if absent). `time` = `signal_time`
    formatted as local 12-hour `DD Mon h:mm AM/PM` (e.g. `04 Jun 1:36 PM`). For a
    prod row this is when `#prod_release` was posted — exactly the "release time".
  - **Line 2:** `<a href="thread_url">subject</a>` (plain subject if no thread url).
  - **Right-side `button`:** the Redmine `status` as a colored chip
    (`statusColor`: green = closed/resolved/tested/done, amber = paused/qa/hold,
    blue = new, purple = reopened, grey = other), `onClick` opens the issue
    (falls back to the thread url).
- **Summary section** (final): a `textParagraph` —
  `<b>Summary</b> — <Label>: n · … · <b>Total: N</b>` over non-empty buckets.
- `reassigned_away` items are not shown in the card (consistent with the report's
  separate handling); they remain in `--json`.

Time is rendered in the runtime's **local timezone** (the user runs in IST);
`signal_time` is stored UTC and converted at format time.

## Architecture

Mirrors the existing pure-logic / thin-IO split.

- **`lib/standup-card.js`** (new, **pure**, no I/O):
  - `buildStandupCard(buckets, meta)` → the `cardsV2` payload object. `meta` =
    `{ title, subtitle }` (caller composes them). Iterates `BUCKET_ORDER`
    (imported from `lib/standup.js`), emits a section per non-empty bucket and the
    summary section. HTML-escapes all interpolated text; builds `<a>` links only
    when the url is present.
  - `statusColor(status)` → `{red,green,blue}` per the mapping above.
  - `formatSignalTime(iso)` → local 12-hour `DD Mon h:mm AM/PM` string (`""` on
    missing/invalid). Uses `Date`/`toLocaleString` so it follows the machine TZ.
  - Exports the three; unit-tested by asserting the returned object's structure
    (section count, headers with counts, row text contains the links/time, status
    button color, empty buckets omitted, escaping) — no network.
- **`lib/chat-api.js`** — add `postToWebhook(webhookUrl, body)`: a plain
  `POST <webhookUrl>` with JSON body via the existing `fetchWithRetry`; **no**
  OAuth/Authorization header (the webhook URL carries its own key+token). Returns
  the parsed response (`{ name, ... }`). This is the only new I/O.
- **`lib/commands.js` `cmdStandup`** — accept `opts.card` and `opts.webhook`.
  After classification/grouping, when `card`: resolve the webhook (see above;
  `fail` if unresolved), build `title`/`subtitle`, `buildStandupCard(...)`,
  `postToWebhook(...)`. Output: human → `Posted standup card to <alias> · <msg>`;
  `--json` → `{ ok, posted: true, message_name, webhook: <alias|"(url)">, count }`.
  The token is never echoed (report the alias, or `(url)` when a raw url was
  passed). The default `hours` becomes 24.
- **`bin/lwchat.js`** — add a `--card` boolean (global-style flag) and pop
  `--webhook <value>`; pass `{ hours, spaceAlias, card, webhook }` to `cmdStandup`.
  Update the default `hours` to 24 and the usage line.
- **`lib/config.js`** — add `standup_webhooks: {}` to `DEFAULT_CONFIG` so the key
  is discoverable/documented (already used at runtime).

## Secret handling

- Webhook URLs only in `~/.lwchat/config.json` (outside the repo; the file holds
  the secret, perms tightened to `600`). Never logged or printed; `--json` and
  human output reference the **alias**, not the URL.
- If a raw `--webhook <url>` is passed, it is not echoed back (shown as `(url)`).

## Testing

**Unit (`lib/standup-card.js`, pure):**
- `buildStandupCard`: section per non-empty bucket with correct `(count)` header;
  empty buckets omitted; row line-1 contains the `#id`→`issue_url` and time, line-2
  the subject→`thread_url`; status button present with the mapped color; summary
  section totals; HTML-escaping of `<`/`&` in subjects; links omitted when urls null.
- `statusColor`: each status family → expected color; unknown → grey.
- `formatSignalTime`: a known UTC instant → expected 12-hour local string;
  empty/invalid → `""`.

**Manual (live, already proven during design):**
- `lwchat standup --card` posts a clickable card to the Daily-Summary webhook;
  ids open Redmine, subjects open threads, status chips colored, time shown.
- Webhook resolution: alias, raw url, and the single-entry default; missing →
  clear error.
- Default window is 24h; `--hours 72` widens it.
- Read path unchanged when `--card` is absent.
